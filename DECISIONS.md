# DECISIONS — Архитектурные решения VS MOTORS

Здесь фиксируются все архитектурные и технические решения проекта.
Формат: решение → причина → компромисс.

---

## Начальные решения (май 2026)

---

### Один владелец без JWT — сравнение OWNER_TELEGRAM_ID
**Решение:** Привилегированный доступ определяется сравнением `telegram_id` из `initData` с `OWNER_TELEGRAM_ID` в ENV. Никакого пароля, никакой базы пользователей, никакого JWT для обычных посетителей.
**Почему:** Приложение принадлежит одному человеку. Любая система логина — избыточна. Telegram сам авторизует владельца через initData.
**Компромисс:** При смене владельца нужно менять ENV переменную и редеплоить. Приемлемо.
**Правило:** Никогда не хранить `OWNER_TELEGRAM_ID` в БД — только в ENV. Значение в коде не захардкоживать.

---

### Обычные пользователи не авторизуются вообще
**Решение:** GET-эндпоинты публичные, без токенов. Пользователи видят объявления без каких-либо запросов авторизации.
**Почему:** Максимальная простота. Нет учётных записей, нет избранного, нет истории. Нечего защищать.
**Компромисс:** Нельзя персонализировать контент. Не нужно на данном этапе.

---

### Без i18n — только русский язык
**Решение:** Все строки интерфейса захардкожены на русском. Никакого i18next, никаких файлов переводов.
**Почему:** Один язык — нет RTL-сложностей, нет накладных расходов на поддержку переводов, быстрее разработка.
**Компромисс:** Добавление языков потребует рефакторинга (обернуть строки в t()). Запланировано как опция для v2.
**Правило:** Не использовать CSS logical properties (padding-inline-start и т.д.) — нет RTL, можно использовать обычные left/right.

---

### asyncpg statement_cache_size=0
**Решение:** Принудительно отключаем кэш prepared statements в asyncpg.
**Почему:** Supabase Session Pooler не поддерживает prepared statements — возникают случайные ошибки в продакшне при использовании pooler-строки подключения.
**Правило:** Всегда указывать `statement_cache_size=0` при подключении к Supabase через pooler. Параметр называется именно `statement_cache_size` (не `prepared_statement_cache_size`).

---

### Cloudinary signed upload
**Решение:** Бэкенд генерирует одноразовую подпись, фронтенд грузит файлы напрямую в Cloudinary.
**Почему:** Не нагружаем бэкенд трафиком фото (Render Free tier имеет лимиты). Cloudinary берёт файл напрямую.
**Компромисс:** Немного сложнее флоу, но экономия на трафике критична на Free tier.
**Правило:** Подпись действительна 60 секунд. Если загрузка не началась за это время — запросить новую.

---

### Валидация фото через `filetype`, не MIME из заголовка
**Решение:** Использовать библиотеку `filetype==1.2.0` (проверка magic bytes), не доверять `Content-Type`.
**Почему:** `Content-Type: image/jpeg` легко подделать — можно передать исполняемый файл. Magic bytes не обманешь.
**Почему не `python-magic`:** требует системную библиотеку `libmagic1`, которая отсутствует на Render Free tier — сервис падает при старте.
**Правило:** Всегда использовать `filetype`, никогда `python-magic` на Render.

---

### Один FastAPI-процесс для API + бота
**Решение:** FastAPI и aiogram 3 запускаются в одном процессе через `lifespan`.
**Почему:** Простота деплоя. Render Free tier даёт один сервис бесплатно — незачем тратить второй на бота.
**Компромисс:** Одна точка отказа. При падении бот и API падают вместе. Приемлемо для проекта одного владельца.
**Когда разделить:** при нагрузке 500+ запросов/мин или если бот начнёт делать тяжёлые операции.

---

### Render Free tier + UptimeRobot keep-alive
**Решение:** Бэкенд на Render Free tier. UptimeRobot пингует `/health` каждые 5 минут.
**Почему:** Бесплатно. Render засыпает через 15 минут неактивности — пинг не даёт уснуть.
**Компромисс:** При первом запросе после сна — задержка 30–60 секунд (cold start). Для личной доски объявлений приемлемо.
**Правило:** UptimeRobot должен быть настроен до первого деплоя в продакшн.

---

### Кнопка «Поделиться» — clipboard, не openTelegramLink
**Решение:** `navigator.clipboard.writeText(url)` с тостом «Ссылка скопирована». Fallback — `tg.showAlert(url)`. URL строится синхронно на клиенте.
**Почему:** `openTelegramLink('https://t.me/bot?startapp=...')` закрывает Mini App и переводит в бот-чат — это навигация, не шаринг. `clipboard.writeText` теряет разрешение браузера если перед вызовом есть `await` — URL нужно строить синхронно.
**Правило:** В Telegram Mini App не использовать `openTelegramLink` для шаринга. Только clipboard + toast.

---

### CORS через ENV переменную
**Решение:** `CORS_ORIGINS` env var — список через запятую; fallback = `MINI_APP_URL + localhost`.
**Почему:** Позволяет добавить preview deployments Vercel без редеплоя бэкенда.
**Правило:** В продакшне явно указывать `CORS_ORIGINS` — не полагаться на fallback.

---

### Контакты владельца — только через ENV, не в БД
**Решение:** Телефон, WhatsApp, Telegram username владельца хранятся в ENV переменных фронтенда (`VITE_OWNER_PHONE`, `VITE_OWNER_WHATSAPP`, `VITE_OWNER_TG_USERNAME`).
**Почему:** Контакты не меняются часто. Хранение в БД — избыточно для одного человека. Изменение = редеплой фронтенда (Vercel делает это за 1 минуту).
**Компромисс:** Нельзя поменять контакты без редеплоя. Приемлемо.

---

### Статусы объявлений: `active` и `sold` (не `deleted`)
**Решение:** Объявления не удаляются физически из БД сразу. Статус `sold` скрывает авто из публичной ленты, но сохраняет запись.
**Почему:** Сохраняется история. Deep links на проданные машины не ломаются — показывается страница со статусом «Продан».
**Исключение:** Физическое удаление (`DELETE`) через AdminPage — явное действие владельца с подтверждением.

---

### Фото хранятся как массив URL в PostgreSQL (TEXT[])
**Решение:** `photos TEXT[]` — массив Cloudinary URL прямо в строке `listings`.
**Почему:** Отдельная таблица `listing_photos` избыточна для одного владельца с ≤20 фото/объявление. Простота запросов.
**Компромисс:** Нельзя делать сложные запросы по фото. Не нужно.
